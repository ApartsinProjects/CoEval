# 3. Methodology

## 3.1 Framework Overview

CoEval is organized as a five-phase pipeline (Figure 1) in which benchmark *construction* and benchmark *scoring* are unified under a single configurable framework. All five phases share a common storage layer with per-item incremental writes, a common model abstraction supporting eight cloud provider interfaces, and a common checkpointing mechanism that enables fine-grained resumption from any point of failure.

The experiments reported in this paper use **ten SOTA API-only models** spanning five providers, each potentially serving as teacher, student, and/or judge within the same experiment (a *dual-role* or *all-role* design). Models are not segregated into separate teacher/student/judge pools; instead, each model's configuration declares all roles it occupies, enabling CoEval to schedule work accordingly. This design maximizes data on model-pair cross-evaluation while minimizing marginal API cost through batch-API discounts (50% for OpenAI and Anthropic; native Gemini pseudo-batching).

CoEval supports two complementary modes for benchmark construction: (1) **generative mode**, in which teacher LLMs produce (prompt, reference-response) pairs from scratch guided by attribute-controlled sampling (Section 3.4.1–3.4.3), and (2) **benchmark-sourced mode**, in which existing public benchmark datasets supply the (prompt, reference-response) pairs directly (Section 3.4.4). Both modes feed into the same Phases 4 and 5, making the evaluation pipeline fully reusable regardless of how the benchmark was constructed.

**Notation.** Let D = {d₁, …, dₙ} denote the *deployment domain* to be evaluated—a natural-language description of the task, the target user population, and any context-specific constraints. Let A = {(aₜ, Vₜ)}ᵢ₌₁ᵏ be the set of *target attributes* (attribute name aₜ, discrete value set Vₜ) and B = {(aₙ, Vₙ)}ⱼ₌₁ᵐ be the set of *nuanced attributes*. Let T = {τ₁, …, τₚ} be the set of teacher models, S = {σ₁, …, σᵧ} be the set of student models, and J = {ξ₁, …, ξᵣ} be the set of judge models.

## 3.2 Phase 1 — Attribute Mapping

**Objective.** Produce a structured decomposition of D into the attribute sets A and B that together span the input space the deployed model will encounter.

**Process.** Each teacher model τᵢ ∈ T receives a structured prompt (Listing 1) containing D and instructions to enumerate:
1. *Target attributes*: dimensions that characterize the type or category of an input (e.g., for a customer service task: `complaint_type`, `urgency_level`, `customer_tier`).
2. *Nuanced attributes*: stylistic and contextual dimensions that modulate input quality (e.g., `formality_register`, `input_length`, `emotional_valence`).

For each attribute, teachers are prompted to propose 3–6 discrete values with short natural-language descriptions of each value. The structured output is parsed from the teacher's response using a typed JSON schema; malformed outputs are retried up to 3 times before being flagged.

When operating in benchmark-sourced mode (Section 3.4.4), attribute sets are inferred automatically from benchmark metadata (e.g., source domain, difficulty tier, problem category) rather than generated by teachers. Phase 1 is skipped entirely when attribute sets are provided statically.

**Merging.** When multiple teachers are used in generative mode, their attribute proposals are merged using a deduplication step: semantically equivalent attributes (identified by embedding similarity > 0.85) are collapsed, with value sets unioned. The merged attribute set is written to `phase1/{task_id}_target_attrs.json` and `phase1/{task_id}_nuanced_attrs.json`.

**Output.** Finalized attribute sets A and B, one JSON file per task.

## 3.3 Phase 2 — Rubric Construction

**Objective.** Produce a domain-specific evaluation rubric R = {(fₗ, wₗ, descₗ)}ₗ₌₁ⁿ where fₗ is a factor name, wₗ ∈ (0, 1] is a relative weight, and descₗ is a natural-language description of what the factor measures.

### 3.3.1 Generative Rubric Mode

In generative mode, each teacher model τᵢ receives D, the merged attribute sets, and a meta-rubric prompt specifying:
- The rubric must include 4–8 named factors.
- Factor names must be machine-readable identifiers (snake_case).
- Factor descriptions must be specific enough that a judge model can apply them consistently without additional context.
- Weights must sum to 1.0.

**Conflict resolution.** If teacher rubrics differ substantially (pairwise cosine similarity of factor-description embeddings < 0.70), a *rubric arbitration* step is triggered: a designated arbitrator teacher reviews all proposals and produces a merged rubric. This occurs in approximately 12% of multi-teacher runs in our experiments.

### 3.3.2 Benchmark-Grounded Rubric Mode

When evaluation datapoints are drawn from an established benchmark dataset, the rubric is derived directly from the benchmark's official evaluation dimensions rather than generated from scratch. This ensures that CoEval's rubric factors are semantically aligned with the criteria the benchmark was designed to measure, enabling a direct comparison between CoEval ensemble scores and the benchmark's native metric.

The mapping from benchmark to rubric factors used in our experiments is:

**Text Summarization (XSum).** The XSum benchmark [55] evaluates summaries along three dimensions captured by the standard ROUGE/BERTScore decomposition: faithfulness to the source (no hallucinated facts), information coverage (key facts from the article are present), and conciseness (the summary is appropriately brief and non-redundant). These map to CoEval rubric factors: `faithfulness` (w=0.40), `information_coverage` (w=0.35), `conciseness` (w=0.25).

**Code Explanation (CodeSearchNet).** CodeSearchNet [56] provides Python function bodies paired with their canonical docstrings, enabling BERTScore-F1 between generated explanations and reference docstrings as the ground-truth metric. CoEval rubric factors: `technical_accuracy` (w=0.35), `explanation_clarity` (w=0.25), `completeness` (w=0.20), `appropriate_level` (w=0.12), `edge_case_handling` (w=0.08).

**Email Composition (AESLC).** The Annotated Enron Subject Line Corpus [58] provides professional email bodies with matched subject lines and metadata, enabling BERTScore-F1 against reference emails as the ground-truth metric. CoEval rubric factors: `clarity` (w=0.25), `appropriate_tone` (w=0.25), `completeness` (w=0.20), `professionalism` (w=0.20), `actionability` (w=0.10).

**Data Interpretation (WikiTableQuestions).** WikiTableQuestions [59] provides Wikipedia table–question–answer triples, enabling exact-match accuracy as the ground-truth metric. CoEval rubric factors: `numerical_accuracy` (w=0.35), `trend_identification` (w=0.25), `insight_quality` (w=0.20), `appropriate_caveats` (w=0.12), `clarity` (w=0.08).

### 3.3.3 Benchmark-Aligned Rubric Dimension

Regardless of whether the rubric is constructed generatively (Section 3.3.1) or derived from a benchmark (Section 3.3.2), each task rubric includes exactly one dimension designated as **benchmark-aligned** and annotated with the tag [BA]. The [BA] dimension is chosen to mechanistically correspond to the benchmark's native evaluation metric, so that a perfect score on [BA] implies alignment with the benchmark's ground-truth signal.

For example:
- **Text Summarization (XSum):** [BA] = `faithfulness` — high scores on this dimension correspond to high BERTScore-F1 against the gold summary, because BERTScore-F1 rewards semantic overlap with the reference, which is the primary operationalization of factual faithfulness.
- **Data Interpretation (WikiTableQuestions):** [BA] = `numerical_accuracy` — this dimension requires judges to verify exact numerical claims against the table, directly paralleling the exact-match accuracy metric used by the benchmark.
- **Code Explanation (CodeSearchNet):** [BA] = `technical_accuracy` — judges verify that the explanation correctly characterizes the function's behavior, aligning with BERTScore-F1 against the canonical docstring.
- **Email Composition (AESLC):** [BA] = `clarity` — the dimension most predictive of BERTScore-F1 against the reference email, as measured by ablation on the calibration set.

The [BA] dimension anchors the rubric to independently validated ground truth while the remaining dimensions (3–4 additional factors) capture quality axes not captured by the benchmark metric alone—such as stylistic appropriateness, edge-case handling, or audience calibration. This hybrid structure is what enables CoEval to both *validate* against benchmark metrics (Section 3.7) and *generalize beyond* them.

**Output.** A single rubric JSON file per task in `phase2/{task_id}_rubric.json`.

## 3.4 Phase 3 — Datapoint Generation

**Objective.** Produce a corpus of N (prompt, reference-response) pairs covering the task distribution that student models will face.

### 3.4.1 Attribute-Controlled Sampling (Generative Mode)

Naïve random generation concentrates on high-frequency attribute combinations and systematically underrepresents rare but deployment-critical scenarios. CoEval addresses this through *stratified attribute sampling*:

1. **Stratum construction.** The Cartesian product V₁ × V₂ × … × Vₖ × Vₙ₁ × … × Vₙₘ defines the full attribute space Ω. Each cell ω ∈ Ω is a stratum.
2. **Budget allocation.** The total datapoint budget N is distributed across strata using a near-uniform allocation with a minimum of 1 datapoint per stratum and a maximum of ⌊N / |Ω|⌋ + δ, where δ is a small integer (typically 2) that allows over-sampling of high-priority strata specified by the user.
3. **Prompt construction.** For each stratum ω = (v₁, …, vₖ, vₙ₁, …, vₙₘ), the teacher receives a generation prompt specifying the exact attribute values and is instructed to produce a (prompt, reference-response) pair that exemplifies those values.

This stratification ensures that all strata are represented, not just the modes of the attribute distribution.

### 3.4.2 Teacher Diversity (Generative Mode)

When |T| > 1, each stratum's quota is divided equally among teachers. Teacher diversity serves two purposes: it reduces surface bias introduced by any single model's stylistic tendencies, and it produces inter-teacher variation that enriches the benchmark by capturing a wider range of prompt phrasings and reference-response styles.

**Bias measurement.** We quantify surface bias using the *attribute coverage ratio* (ACR):
$$\text{ACR} = \frac{1}{|\Omega|} \sum_{\omega \in \Omega} \mathbf{1}[\text{count}(\omega) \geq 1]$$

A perfect benchmark has ACR = 1.0; naive uncontrolled generation typically achieves ACR ≈ 0.40–0.55 for attribute spaces with |Ω| > 200 strata.

### 3.4.3 Quality Filtering (Generative Mode)

Generated (prompt, reference-response) pairs are subject to automatic quality checks:
- **Attribute adherence**: a lightweight classifier (fine-tuned DeBERTa-v3-base) verifies that the generated prompt exhibits the specified target attribute values. Pairs with adherence score < 0.80 are regenerated.
- **Reference length**: reference responses shorter than 50 tokens or longer than 1,024 tokens are flagged for human review.
- **Duplication**: near-duplicate detection (MinHash LSH, Jaccard threshold 0.70) removes redundant pairs.

**Output.** A JSONL file `phase3/{task_id}_{teacher_id}.jsonl`, one line per (prompt, reference) pair, including the attribute assignment for each pair.

### 3.4.4 Benchmark-Sourced Datapoints

When a public benchmark dataset is available for the target task, CoEval can draw (prompt, reference-response) pairs directly from the benchmark's held-out evaluation split, bypassing teacher generation entirely. This mode serves two purposes: (1) it provides a reproducible, independently validated source of evaluation items, and (2) it enables direct measurement of whether CoEval ensemble scores track the benchmark's own evaluation metric—a stronger form of validation than comparing to a separately collected annotation set.

**Integration protocol.** A benchmark loader transforms each benchmark item into CoEval's standard (prompt, reference-response, attribute-assignment) record:
- The benchmark's *input* (source document, code snippet, chart description, or email context) becomes the `prompt`.
- The benchmark's *gold-standard output* (reference summary, canonical solution, gold answer, or reference email) becomes the `reference_response`.
- Benchmark metadata fields (domain, difficulty tier, problem category) are mapped to CoEval attribute values via a per-dataset schema file.

**Stratified sampling from benchmarks.** The attribute-controlled sampling of Section 3.4.1 applies unchanged: pairs are stratified across the inferred attribute values, ensuring that the selected benchmark items cover the benchmark's difficulty and domain distribution rather than sampling from its mode.

**YAML configuration.** Benchmark-sourced mode is activated via the `phase3` block in the experiment config:

```yaml
phase3:
  source: benchmark
  benchmark:
    dataset: xsum          # Registered dataset name
    split: validation
    sample_size: 620       # Items to draw (stratified)
    attribute_map: benchmark/xsum_attribute_map.yaml
```

When `source: benchmark` is set, Phases 1 and 2 are automatically switched to `Keep` mode (attributes and rubric are imported from the benchmark schema rather than generated), and Phase 3 skips teacher-model API calls entirely—reducing pipeline cost proportionally to the number of teachers that would otherwise have been used.

**Output.** A single JSONL file `phase3/benchmark_{dataset_id}.jsonl` following the same schema as teacher-generated files, enabling Phases 4 and 5 to operate identically regardless of sourcing mode.

## 3.5 Phase 4 — Response Collection

**Objective.** Collect responses from each student model σ ∈ S to each benchmark prompt.

**Process.** Each student model σ receives the prompt *without* the reference response. Responses are generated with temperature T = 0.0 (greedy decoding) to ensure reproducibility. Each (prompt, student-response) pair is assigned a unique `response_id` and written to `phase4/{task_id}_{student_id}.jsonl` immediately upon generation.

**Skip logic.** Phase 4 operates in `Extend` mode by default: at the start of each run, the set of already-responded datapoint IDs is loaded from existing JSONL files. Only missing IDs are sent to the student model. This ensures that a crashed run wastes only the partially-generated batch.

**Batch API.** For cloud providers that support native batch processing (OpenAI Batch API; Anthropic Message Batches API), CoEval submits all Phase 4 requests for a given provider as a single batch job, achieving a 50% cost reduction relative to synchronous calls. Batch status is polled automatically; the `coeval status` subcommand provides a live dashboard.

**Output.** A JSONL file per (task, student) pair. Each record contains: `datapoint_id`, `student_id`, `prompt`, `response`, `token_count` (approximate token length of the response, used for verbosity-bias analysis), `teacher_id` or `benchmark_id`, `attribute_assignment`.

## 3.6 Phase 5 — Ensemble Scoring

**Objective.** Produce a multi-dimensional quality score vector **q** ∈ [1, 5]ⁿ for each (prompt, reference, student-response) triple, using a heterogeneous ensemble of judge models.

### 3.6.1 Scoring Protocol

Each judge ξ ∈ J receives a structured scoring prompt containing:
1. The deployment domain description D.
2. The rubric R with factor descriptions and weights.
3. The benchmark prompt.
4. The reference response (labeled as "reference answer").
5. The student response (labeled as "response to evaluate").

The judge is instructed to score the student response on each rubric factor on a 1–5 integer scale, provide a one-sentence justification per factor, and output a valid JSON object. The prompt explicitly prohibits sycophantic response patterns by instructing the judge to score *relative to the reference*, not in absolute terms.

### 3.6.2 Ensemble Aggregation

Let qₗⱼ ∈ {1, 2, 3, 4, 5} be judge ξⱼ's score on factor fₗ. The ensemble score for factor fₗ is computed as the *weighted median*:
$$\hat{q}_l = \text{weightedMedian}(\{q_{lj}\}_{j=1}^{|J|}, \{c_j\}_{j=1}^{|J|})$$
where cⱼ is a *judge calibration weight* (described below). The weighted median is used rather than the mean because it is robust to outlier judges who assign extreme scores due to positional bias or format mismatches.

The composite score for a student response is:
$$Q = \sum_{l=1}^{n} w_l \cdot \hat{q}_l$$

**Uncertainty estimation.** For each factor, we report the inter-judge standard deviation σₗ as a measure of scoring uncertainty. Responses with σₗ > 1.5 on any rubric factor are flagged as *high-uncertainty* and deprioritized in ranking comparisons.

### 3.6.3 Judge Calibration

Judge models differ in their calibration: some are systematically lenient (prefer high scores) while others are strict. Raw scores are calibrated using a two-parameter linear model:
$$q_{lj}^* = \alpha_j \cdot q_{lj} + \beta_j$$
where αⱼ and βⱼ are estimated on a *calibration set* of 200 (prompt, reference, response) triples drawn from benchmark validation splits with known benchmark-metric scores (ROUGE-L for summarization, pass@1 for code, exact-match for data interpretation, BERTScore-F1 for email composition). The calibration mapping aligns judge ordinal scores with the benchmark's continuous metric, ensuring that the ensemble's composite score Q is monotonically consistent with the benchmark's own evaluation signal.

### 3.6.4 Positional Bias Mitigation

For each (prompt, reference, student-response) triple, each judge evaluates it in two orderings: (reference, student-response) and (student-response, reference). The judge's scores from both orderings are averaged, neutralizing positional preference. This doubles the scoring cost but eliminates the ordering confound documented by [6, 23].

**Output.** A JSONL file `phase5/{task_id}_{student_id}.jsonl`. Each record contains: `response_id`, per-factor raw scores from each judge, calibrated ensemble scores, uncertainty estimates, and the final composite score Q.

## 3.7 Benchmark Validation Protocol

To assess the reliability of CoEval ensemble scores, we measure their alignment with benchmark-native ground-truth metrics. For each task, we compute the Spearman rank correlation ρ between the CoEval composite score Q assigned to each student response and the benchmark-native score for the same item:

| Task | Benchmark | Ground-Truth Metric |
|------|-----------|-------------------|
| Text Summarization | XSum [55] | BERTScore-F1 against gold summary |
| Code Explanation | CodeSearchNet [56] | BERTScore-F1 against reference docstring |
| Email Composition | AESLC [58] | BERTScore-F1 against reference email |
| Data Interpretation | WikiTableQuestions [59] | Exact-match accuracy |

The benchmark upper bound reported in Table 3 is the intra-benchmark agreement ceiling: for XSum and email composition this is the test-retest BERTScore reliability of the gold reference (ρ ≈ 0.89–0.92 across runs with different decoding seeds); for HumanEval this is the agreement between pass@1 and pass@10 as a consistency check; for ChartQA it is the human ceiling reported in the original paper.

This protocol makes CoEval's reliability claims fully reproducible: any researcher with access to the public benchmark datasets and the CoEval codebase can independently verify the reported correlation figures.

## 3.8 Dual-Track Validation

To cross-validate that student rankings are stable across data sources rather than an artifact of any particular datapoint collection strategy, CoEval evaluates each student on two parallel tracks:

**Track A (benchmark-sourced).** Evaluation datapoints are drawn from the held-out splits of public benchmark datasets using the `benchmark` virtual interface (Section 3.4.4). Because these items are independently validated and their ground-truth labels are determined by established benchmark protocols, Track A scores serve as the primary reliability reference.

**Track B (synthetic).** Evaluation datapoints are generated by teacher LLMs (GPT-4o, GPT-4o-mini, Gemini 2.0 Flash, and Llama 3.3 70B in the reported experiments) using the attribute-controlled sampling procedure of Section 3.4.1. Track B scores reflect a student's capability on LLM-generated prompts, which may have different distributional properties (stylistic, difficulty) than benchmark items.

**Stability measurement.** For each task, we compute Kendall's τ between the student rankings produced by Track A composite scores and Track B composite scores. High τ (≥ 0.85) indicates that CoEval's rankings are robust to the data source: the same models rank highly regardless of whether items come from curated benchmark splits or from synthetic generation. Divergence (τ < 0.70) flags task-specific data-source sensitivity and is reported alongside the main results.

This dual-track design serves a second purpose: it separates the *evaluation pipeline's reliability* (verified by Track A vs. benchmark-native metrics) from the *benchmark's coverage* (verified by Track B's generative diversity). Both dimensions are necessary for a benchmark framework to be trustworthy.

## 3.9 Fault Tolerance and Resumption

A central engineering goal of CoEval is the ability to resume any interrupted run with minimal repeated work. The system implements two layers of fault tolerance:

**Phase-level resumption.** The `meta.json` file in the experiment root tracks `phases_completed`, `phases_in_progress`, and a per-phase `started_at`/`completed_at` timestamp. Phases listed in `phases_completed` are skipped entirely on restart.

**Item-level resumption.** Within phases 3–5, each completed item is appended to a JSONL file immediately upon generation (rather than batched). On restart (`--continue` flag), the set of already-written item IDs is loaded and only missing items are processed. This granularity means that a crash after generating 847 out of 1,000 datapoints wastes at most one in-flight batch (typically 8–16 items) rather than the full 847.

The `--continue` CLI flag activates both layers simultaneously:
```bash
coeval run --config experiment.yaml --continue
```

## 3.10 Configuration Interface

CoEval is configured via a declarative YAML file. Listing 2 shows a generative-mode experiment; Listing 3 shows the dual-track 10-model all-roles configuration used in the reported experiments.

**Listing 2: Generative mode (teacher-generated datapoints)**
```yaml
experiment:
  id: "customer-service-eval-v1"
  storage_folder: "./experiments"

domain:
  name: "Customer service email response"
  description: |
    Evaluate LLM responses to customer service email queries
    for a mid-market SaaS company.

teachers:
  - model: claude-opus-4-6
    provider: anthropic
  - model: gpt-4o
    provider: openai

students:
  - model: claude-sonnet-4-6
    provider: anthropic
  - model: llama-3-70b-instruct
    provider: huggingface

judges:
  - model: claude-opus-4-6
    provider: anthropic
  - model: gpt-4o
    provider: openai
  - model: gemini-1.5-pro
    provider: google

generation:
  datapoints_per_task: 200
```

**Listing 3: Dual-track 10-model all-roles configuration with `interface: auto`**
```yaml
experiment:
  id: "dual-track-10model-v1"
  storage_folder: "./experiments"

domain:
  name: "Multi-task NLP evaluation"
  description: |
    10-model dual-track evaluation across four NLP tasks.
    Track A draws items from public benchmark datasets;
    Track B uses LLM-generated synthetic datapoints.

models:
  # --- Frontier API models (multi-role) ---
  - name: gpt-4o
    interface: openai
    roles: [teacher, student, judge]
  - name: gpt-4o-mini
    interface: openai
    roles: [teacher, student, judge]
  - name: claude-sonnet-4-6
    interface: anthropic
    roles: [student, judge]
  - name: claude-haiku-3-5
    interface: anthropic
    roles: [student, judge]
  - name: gemini-2.0-flash
    interface: gemini
    roles: [teacher, student, judge]

  # --- Open-weight models via OpenRouter ---
  - name: llama-3.3-70b-instruct
    interface: openrouter
    roles: [teacher, student]
  - name: llama-3.1-8b-instruct
    interface: openrouter
    roles: [student]
  - name: mistral-small-24b
    interface: openrouter
    roles: [student]
  - name: deepseek-v3
    interface: auto           # CoEval selects cheapest provider automatically
    parameters:
      model: deepseek/deepseek-chat
    roles: [student]
  - name: qwen-2.5-72b
    interface: openrouter
    roles: [student]

  # --- Track A: benchmark virtual teacher ---
  - name: benchmark_data
    interface: benchmark
    roles: [teacher]

generation:
  datapoints_per_task: 100
  tracks: [A, B]             # A = benchmark-sourced; B = synthetic
```

**`interface: auto`.** Setting `interface: auto` on a model causes CoEval to automatically select the cheapest available provider for that model at runtime, consulting the routing table in `benchmark/provider_pricing.yaml`. The routing table maps model identifiers to provider costs per million tokens; CoEval picks the provider with the lowest effective cost for which valid credentials are present in the key file. This is particularly useful for models available on multiple platforms (e.g., DeepSeek V3 on both OpenRouter and a direct endpoint) where the optimal routing may change over time.

## 3.11 Implementation Details

CoEval is implemented in Python 3.11+ with no mandatory dependencies beyond the standard library and `pyyaml`. Optional dependencies (`openai`, `anthropic`, `google-generativeai`, `transformers`, `torch`, `datasets`) are declared as extras and installed only for the providers and benchmark loaders in use. The codebase comprises approximately 5,200 lines of source code and 2,100 lines of tests (pytest, 483 tests), achieving 94% line coverage.

**Concurrency.** Phases 3–5 use asyncio-based concurrency with a configurable maximum of concurrent in-flight API requests (default: 8 per model). Rate limiting is implemented with a token-bucket algorithm calibrated to each provider's documented tier limits.

**Benchmark loaders.** Each supported benchmark dataset has a registered loader class that (a) downloads or reads the dataset via the HuggingFace `datasets` library, (b) applies the attribute mapping, and (c) emits CoEval-compatible JSONL records. Loaders are registered via a simple entry-point mechanism, making it straightforward to add support for new benchmarks.

**Reproducibility.** All generated artifacts include the model name, temperature, and prompt hash, enabling exact reproduction of any run. Experiments are versioned with a schema version field in `meta.json`; future schema migrations include a backward-compatible reader.
