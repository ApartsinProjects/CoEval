## Appendix A: Table of Mathematical Notation

| Symbol | Definition |
|--------|------------|
| T | Set of teacher model identifiers |
| S | Set of student model identifiers |
| J | Set of judge model identifiers |
| τ | Task identifier |
| A_τ | Set of target attributes for task τ |
| Ω_τ | Full attribute stratum space for task τ (Cartesian product of A_τ values) |
| d_i | Datapoint i: (prompt p_i, reference response r_i, attributes ω_i) |
| D | Set of all datapoints across all tasks and teachers |
| D_τ,t | Datapoints for task τ generated by teacher t |
| e_{s,i} | Response of student model s to prompt p_i |
| f | Rubric factor (named scoring dimension, e.g., "accuracy") |
| F_τ | Set of rubric factors for task τ |
| score(j, s, i, f) | Score ∈ {High, Medium, Low} assigned by judge j to student s on datapoint i, factor f |
| score_norm | Normalised score: High=1.0, Medium=0.5, Low=0.0 |
| Q(s) | Composite quality score for student s: mean score_norm over all valid (s, i, f) units |
| ACR_τ | Attribute Coverage Ratio for task τ: fraction of Ω_τ cells with ≥1 datapoint |
| RAR_τ | Rare-Attribute Recall for task τ: ACR over strata with natural frequency < 3 |
| PFR(j) | Positional Flip Rate for judge j: fraction of comparisons changed by order reversal |
| ρ | Spearman rank correlation coefficient |
| τ_K | Kendall rank correlation coefficient |
| J* | Selected judge subset (high-agreement ensemble) for robust filtering |
| T* | Selected teacher subset (high-differentiation) for robust filtering |
| D* | Robust datapoints: produced by T*, scored by J* with consistency ≥ θ |
| θ | Agreement consistency threshold for robust filtering (default: 1.0) |
| α_j, β_j | Calibration parameters for judge j (shift and scale) |

## Appendix B: Example Rubric Definitions

The following table shows the per-task rubric factors used in the benchmark-sourced experiment configuration. These factors were derived from the benchmark datasets' official evaluation criteria and validated in Phase 2 of the CoEval pipeline.

**Text Summarization (XSum)**

| Factor | Definition |
|--------|------------|
| `faithfulness` | All factual claims in the summary are supported by the source article; no hallucination. |
| `coverage` | The summary captures all major points present in the source without omission. |
| `conciseness` | The summary is free of redundancy and remains within the specified length. |
| `fluency` | The text is grammatically correct, well-structured, and reads naturally. |
| `alignment_with_reference` | The summary conveys similar information and emphasis to the gold reference summary. |

**Code Explanation (CodeSearchNet)**

| Factor | Definition |
|--------|------------|
| `technical_accuracy` | All statements about what the code does are factually correct. |
| `explanation_clarity` | The explanation avoids unnecessary jargon for the specified audience level. |
| `completeness` | All significant code paths and edge cases are addressed without omission. |
| `appropriate_level` | Depth and vocabulary match the specified audience (beginner / intermediate / advanced). |
| `practical_value` | The explanation helps the reader understand when and why to use this pattern. |

**Email Composition (AESLC)**

| Factor | Definition |
|--------|------------|
| `clarity` | The email communicates its purpose unambiguously in the opening paragraph. |
| `appropriate_tone` | Tone and formality match the specified relationship and context. |
| `completeness` | All necessary information is included; no critical detail is omitted. |
| `professionalism` | The email follows professional conventions and is free of grammatical errors. |
| `actionability` | Where a response is expected, the request is specific and easy to act on. |

**Data Interpretation (ChartQA)**

| Factor | Definition |
|--------|------------|
| `accuracy` | All numerical claims and trend descriptions correctly reflect the data shown. |
| `insight_quality` | The interpretation goes beyond restating the obvious; explains why the trend matters. |
| `statistical_literacy` | Statistical language is used correctly and proportionally to the data complexity. |
| `clarity` | The response is structured: observation → explanation → implication. |
| `appropriate_caveats` | Limitations of the data are acknowledged where relevant; claims are not overstated. |

## Appendix C: Full YAML Configuration Example

The following YAML configuration implements the benchmark-sourced experiment described in Section 4.1. Teachers use the `benchmark` virtual interface (Phase 3 is automatically skipped); students and judges use the OpenAI API with Batch API enabled.

```yaml
# CoEval benchmark-sourced experiment (abbreviated — see benchmark/mixed.yaml for full)
models:
  - name: benchmark
    interface: benchmark
    parameters: {}
    roles: [teacher]

  - name: gpt-4o
    interface: openai
    parameters:
      model: gpt-4o
      temperature: 0.0
      max_tokens: 512
    roles: [student, judge]
    role_parameters:
      student:
        temperature: 0.7
        max_tokens: 256
      judge:
        temperature: 0.0
        max_tokens: 128

  - name: claude-3-5-sonnet-20241022
    interface: anthropic
    parameters:
      model: claude-3-5-sonnet-20241022
      temperature: 0.7
      max_tokens: 512
    roles: [student, judge]

tasks:
  - name: text_summarization
    description: >
      Summarise a given passage of text, capturing key information at the
      specified depth with the appropriate tone for the target audience.
    output_description: >
      A summary of 50–150 words in plain prose. No bullet points unless
      explicitly requested.
    target_attributes:
      complexity: [simple, moderate, complex, technical]
      domain:     [science, business, politics, technology, health]
    nuanced_attributes: {}
    sampling:
      target: [1, 2]
      nuance: [0, 0]
      total: 620
    rubric:
      faithfulness:  "All claims in the summary are supported by the source."
      coverage:      "All major points are captured without omission."
      conciseness:   "The summary avoids redundancy and respects the length limit."
      fluency:       "The text is grammatically correct and reads naturally."
    evaluation_mode: single

experiment:
  id: benchmark-sourced-v1
  storage_folder: ./benchmark/runs
  log_level: INFO
  phases:
    attribute_mapping:   Keep     # static from attribute_map.yaml
    rubric_mapping:      Keep     # static from task YAML above
    data_generation:     Extend   # benchmark data pre-ingested; no generation needed
    response_collection: New
    evaluation:          New
  batch:
    openai:
      response_collection: true
      evaluation: true
  quota:
    gpt-4o: {max_calls: 5000}
    claude-3-5-sonnet-20241022: {max_calls: 5000}
```
